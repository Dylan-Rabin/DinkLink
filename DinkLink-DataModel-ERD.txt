======================================================================
  GPN INTEGRATION NOTES
======================================================================

  GPN HAS a real developer REST API at globalpickleball.network/developers.
  Registration as a developer is required. Key available endpoints:

    GET  /user/calculated-levels   ← singles and doubles playing levels
    GET  /user/stats               ← match history stats
    GET  /user/info                ← profile info
    POST /login                    ← get access token (OAuth-style)

  Recommended integration approach:

  1. User links their GPN account via an in-app OAuth / login flow.
  2. DinkLink Supabase Edge Function: gpn-oauth handles the token exchange.
     Resulting GPN access_token stored securely in Supabase Vault
     (NOT in user_profiles — tokens are server-side only).
  3. On sync trigger (user taps Refresh or daily background job):
     Edge Function calls GET /user/calculated-levels and GET /user/stats
     using the stored token.
  4. Edge Function writes gpn_singles_level, gpn_doubles_level,
     gpn_dupr_rating, gpn_last_synced_at back to user_profiles.
  5. App reads and displays cached values from user_profiles.

  Offline behaviour: Show cached values with "Last updated X days ago".
  GPN sync button disabled when NetworkMonitor.isConnected = false.
  Show stale warning if gpn_last_synced_at > 7 days ago.

  DUPR: GPN is a preferred DUPR partner. DUPR ratings appear on GPN
  profiles and are returned by the calculated-levels endpoint.


======================================================================
  KEY DESIGN DECISIONS
======================================================================

  1. level_tiers is kept as Swift constants, NOT a DB table (Option A)
     — Tier thresholds are stable. Keeping them in ProgressionService.swift
       avoids an unnecessary table, join, and cache strategy.
     — 4 ranks per MVP: Bronze Paddle | Silver Spin | Gold Rally | Diamond Dink

  2. user_progression is merged into user_profiles (Option A)
     — Both were 1:1 with the user. Merging removes a table and a join.
     — total_xp and xp_updated_at are added directly to user_profiles.

  3. XP sources match MVP spec (not current ProgressionService.swift)
     — Complete session +50, 10+ clean hits +20, Personal best +40,
       Friend match +30, 5-day streak +75, Challenge win +100.
     — streak_tracking requires current_streak / last_active_date columns.

  4. shot_events is a remote table (new, from MVP spec)
     — MVP data model section 6 explicitly includes ShotEvent with
       gameId and sessionId as a persisted model.
     — Enables future biomechanics analysis and shot-level replays.

  5. GPN uses its real OAuth API, not HTML scraping
     — GPN developer portal exposes Get Login Token, Get User's
       Calculated Levels, Get User's Stats.
     — Tokens handled server-side in Supabase Vault via Edge Function.

  6. badges catalog is server-side
     — App fetches the full list to show locked/unlocked state.
       New badges require no Swift code changes.

  7. saved_locations replaces the single locationName string
     — Supports Home + multiple favorite courts.
     — Latitude/longitude enables court-proximity and weather features.

  8. user_profiles is separate from auth.users
     — Supabase best practice. Keeps auth isolated from app profile data.
